# Atropos — Agent Reference Guide

This file is the agent-facing contract for the Atropos codebase. Read it first before making changes. For deeper implementation details, consult `ARCHITECTURE.md`.

## Non-Negotiable Rules

- `originalImage` is immutable after load.
- `SaveImage` reads only `warpedImage`.
- `workingImage()` must be used for pre-warp paths; do not read `warpedImage` directly.
- The four modes are mutually exclusive; they are not a pipeline.
- Any reset/load path that clears image state must also cancel in-flight touch-up.
- Any reset/load path that clears image state must also null `cornerEntryRef` (see `resetImageState`).
- Leaving corner mode must call `CancelCornerDetect()` and increment `detectGenRef` before any await.
- Any committing operation must preserve undo behavior and mode invariants.
- Do NOT edit `frontend/wailsjs/go/main/App.js` by hand. JS bridge files in `frontend/wailsjs/go/` are automatically regenerated by Wails (for example via the usual `wails dev` or build steps).

## Technology Stack

- Backend: Go, Wails v2
- Frontend: React (JSX), plain CSS
- Image processing: pure Go; no OpenCV; no external image libraries except `golang.org/x/image` for BMP/TIFF encode/decode
- Wails bridge: `frontend/wailsjs/go/main/App.js` and `frontend/wailsjs/go/models.ts`

## Frontend Layout

`App.jsx` is a thin coordination layer. Most logic lives under `frontend/src/`.

Key files:
- `hooks/useImageActions.js` — Go action handlers, mode switching, load/save/reset orchestration
- `hooks/useMouseHandlers.js` — all pointer interaction
- `hooks/useKeyboardShortcuts.js` — keyboard shortcuts and browser-behavior suppression
- `hooks/useTouchup.js` — touch-up brush state machine
- `hooks/useZoomPan.js` — viewport, zoom, pan, fit-width logic
- `hooks/usePersistentSettings.js` — file-backed settings and migration
- `hooks/useStatusMessage.js` — status text and fade timer
- `components/ImageOverlays.jsx` — all SVG overlays
- `components/CompositorModal.jsx` — image stitching UI
- `frontend/wailsjs/go/main/App.js` — autogenerated bridge surface (do not edit; regenerate via the Wails toolchain)

## Mode Model

Corner, Disc, Line, and Normal modes are independent entry points into the same scan. Switching modes always resets warp state. A user may crop one object, save, then switch to a different mode on the same scan.

Mode switching must:
1. cancel any in-flight touch-up
2. reset the mode being left
3. restore cached corner state when valid
4. otherwise auto-detect or refresh preview as appropriate

## Critical Image State

Understand the image fields before editing anything:

- `originalImage` — immutable source image
- `currentImage` — pre-warp working image
- `warpedImage` — committed result; this is what `SaveImage` writes
- `levelsBaseImage` — snapshot for non-stacking levels edits
- `discBaseImage` — deterministic source for disc redraws
- `discWorkingCrop` — cached subregion used by disc redraws
- `undoStack` — capped LIFO undo history

## Common Pitfalls

1. Reading `warpedImage` directly instead of `workingImage()` breaks pre-warp paths.
2. Forgetting `saveUndo()` before a committing operation makes it non-undoable.
3. Calling `saveUndo()` inside `SetLevels` floods the undo stack.
4. Not clearing `warpedImage` in a reset path leaves stale warp state behind.
5. Adding a Go method without updating `App.js` causes undefined bridge calls.
6. Using `el.offsetLeft/offsetTop` for persistent overlays is wrong; use the SVG overlay system.
7. Setting `fitWidth` to 0 and waiting for `onLoad` during mode switch can cause the overlay to flood the canvas.
8. Using IOPaint for warp out-of-bounds fill is wrong; PatchMatch is used for `applyWarpFill`.
8a. `iopaintFill` does NOT send the full source image to the IOPaint server. It crops to the bounding box of the mask (+ 128 px margin), sends that crop as JPEG, then composites only the masked pixels from the response back onto a full clone of the source. The function always returns a full-size `*image.NRGBA`. Do not assume the returned image reflects IOPaint's view of the whole image.
9. Forgetting `e.preventDefault()` for repeated keydown events allows browser behavior to fire on every repeat.
10. Not clearing frontend-only selection state in `handleSkipCrop` leaves stale overlays visible.
11. Omitting disc state from `LoadImage` / `RecropImage` resets can leak stale disc state into later operations.
12. Forgetting to reset tool toggles in `loadFile` leaves tools visually active on the new image.
13. Forgetting `cancelTouchup()` in a new reset or load path allows an in-flight touch-up to mutate cleared state.
14. Disc translation must honor current disc rotation in both live preview and backend commit (use a shared transformation path and keep `discRotation` in sync).
15. Not clearing `cornerEntryRef` in `resetImageState` causes the previous image's preview to flash back when loading a new image while in corner mode.
16. Calling `setMode(m)` after async operations instead of at the top of `handleModeSwitch` causes mode buttons to lag behind all other visual changes.
17. Setting `overflow: hidden` on the image wrapper div (a flex item inside `.canvas-area`) silently reduces its `min-width: auto` to 0 per the CSS flexbox spec, which allows flexbox to shrink the wrapper to fit the canvas and eliminates horizontal scroll entirely. Always pair `overflow: hidden` on that element with `flex-shrink: 0`.

## Where to Look

- `app.go` — top-level state, startup, orchestration
- `app_adjust.go` — crop, rotate, levels, auto-contrast, trim borders, undo, working image helpers
- `app_mode.go` — clean preview behavior
- `app_io.go` — load, save, recrop, file I/O
- `app_corner.go` — corners, warp fill, perspective operations
- `app_disc.go` — disc rendering
- `app_normal.go` — normal rectangle crop
- `app_line.go` — line mode / quadrilateral crop
- `app_touchup.go`, `app_iopaint.go`, `patchmatch.go` — touch-up pipeline
- `app_compositor.go`, `compositor.go` — stitching feature
- `frontend/src/hooks/*` — all frontend interaction logic
- `frontend/wailsjs/go/main/App.js` — bridge surface that must stay in sync with Go exports

For precise invariants, field semantics, and operation ordering, read `ARCHITECTURE.md`.
